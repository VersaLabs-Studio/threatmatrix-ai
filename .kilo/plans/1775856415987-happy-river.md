# End‑to‑End Verification Plan

## Objective
Verify that the UI pages, API endpoints, and functional flows listed in the checklist work correctly from a user’s perspective. The plan assumes a JavaScript/TypeScript front‑end (React/Vue/etc.) with an API backend and that Cypress is the chosen e2e framework (replace with Playwright or Selenium if needed).

---

## Prerequisites
- **Local development environment** with `node` (>=16) and `npm`/`yarn` installed.
- Backend services running (e.g., `npm run start:api` or Docker compose) and reachable at `http://localhost:8000` (adjust host/port as required).
- Front‑end dev server running (`npm run dev` or similar) and reachable at `http://localhost:3000`.
- A test user with appropriate roles (ML Ops, Admin, Forensics, etc.) and a valid authentication token.
- **Cypress** installed (`npm i -D cypress`). If you use another framework, replace Cypress commands with the equivalent.
- Directory `cypress/e2e/` (or `tests/e2e/`) ready for test files.
- Sample data for PCAP upload (e.g., `cypress/fixtures/sample.pcap`).

---

## High‑Level Steps
1. **Configure Cypress** – set `baseUrl`, add token handling, and configure file download folder.
2. **Write individual test specs** for each checklist item.
3. **Run the suite** locally and in CI, capture screenshots and videos.
4. **Validate results** – ensure UI elements are present, API responses have expected shape, and side‑effects (PDF download, analysis output) occur.
5. **Report** – generate a JUnit XML or HTML report for CI consumption.

---

## Detailed To‑Do List

- [ ] **Environment Setup**
  - Install Cypress and ensure `cypress.json` (or `cypress.config.ts`) includes:
    ```json
    {
      "baseUrl": "http://localhost:3000",
      "env": { "authToken": "<TOKEN>" },
      "downloadsFolder": "cypress/downloads"
    }
    ```
  - Add a `cypress/support/commands.js` file with a custom `login()` command that injects the token into local storage or cookies.
  - Verify backend services are up: `curl http://localhost:8000/api/health` returns 200.

- [ ] **Test 8 – ML Ops – Load Page & Model Metrics**
  - **Navigate** to ML Ops page (`/ml-ops`).
  - **Intercept** `GET /api/v1/ml/models` and assert status 200 and JSON schema containing `accuracy` and `f1` fields.
  - **Assert** that the UI displays the accuracy and F1 values (e.g., contains text `Accuracy:` and `F1:`).

- [ ] **Test 9 – ML Ops – View Confusion Matrix**
  - **Navigate** to the Confusion Matrix view (might be a tab or modal).
  - **Intercept** `GET /api/v1/ml/metrics` and verify the response includes a `confusionMatrix` array or object.
  - **Assert** that the rendered matrix matches the returned data (check number of rows/columns, cell values).

- [ ] **Test 10 – Reports – Generate PDF Report**
  - **Navigate** to Reports page.
  - **Click** the *Generate Report* button.
  - **Intercept** `GET /api/v1/reports/threat-summary` and verify `Content‑Type: application/pdf`.
  - **Assert** that a file named `threat-summary*.pdf` appears in `cypress/downloads/` and its size > 0.

- [ ] **Test 11 – Intel Hub – Load Page & IOC Correlation**
  - **Navigate** to Intel Hub (`/intel-hub`).
  - **Intercept** `GET /api/v1/intel/iocs` and assert the payload contains an array of IOC objects with expected fields (e.g., `type`, `value`).
  - **Assert** that the UI table/list displays at least one IOC and that correlation visualisation (graph/chart) renders.

- [ ] **Test 12 – Admin – Load Audit Log**
  - **Navigate** to Admin Audit Log page (`/admin/audit-log`).
  - **Intercept** `GET /api/v1/admin/audit-log` and verify each entry includes a `timestamp` property in ISO format.
  - **Assert** that the UI table lists events with human‑readable timestamps (e.g., `2024‑03‑15 12:34:56`).

- [ ] **Test 13 – Forensics – Upload PCAP**
  - **Navigate** to Forensics upload page.
  - **Use** Cypress `cy.fixture('sample.pcap')` and `cy.get('input[type="file"]').attachFile('sample.pcap')`.
  - **Intercept** `POST /api/v1/capture/upload-pcap` and verify a `202 Accepted` (or `200`) and a JSON response containing an `analysisId`.
  - **Assert** that the UI shows an *Analysis results* section populated with data tied to the `analysisId`.

- [ ] **Test 14 – Hunt – Search Flows**
  - **Navigate** to Hunt page.
  - **Enter** a sample query into the search box and submit.
  - **Intercept** `POST /api/v1/flows/search` (or `GET` if implemented that way) and assert a non‑empty `results` array.
  - **Assert** that the results table/grid displays at least one flow matching the query.

- [ ] **Common Assertions & Cleanup**
  - Ensure each test logs out or clears local storage at the end to avoid state leakage.
  - Add `afterEach` hook to take a screenshot on failure.
  - Optionally stub the API responses for faster CI runs, but keep a “full‑stack” test that hits the real backend.

- [ ] **Run the Suite**
  - Execute `npx cypress run --spec "cypress/e2e/**/*.spec.js"` locally.
  - In CI, add a step that runs the same command and uploads the generated JUnit report (`cypress/results/*.xml`).

- [ ] **Validate Results**
  - Review Cypress Dashboard (or HTML report) for passed/failed tests.
  - Verify that PDF download and PCAP analysis UI updates are present in the artifacts.
  - If any test fails, capture the network request/response and console logs to pinpoint the cause.

---

## Optional Extensions
- **Performance checks** – add `cy.wait` assertions to ensure API latency is under a threshold (e.g., < 500 ms).
- **Accessibility** – run `cypress-axe` after each page load to catch a11y violations.
- **Cross‑browser** – configure Cypress to run in Chrome, Edge, and Firefox for broader coverage.

---

## Acceptance Criteria
- All seven checklist items are covered by automated e2e tests.
- Tests run head‑less locally (`cypress run`) and in CI without manual intervention.
- Each test validates both the UI rendering and the underlying API contract.
- PDF download and PCAP upload flows are verified end‑to‑end (real file creation, real analysis result).
- CI pipeline fails if any of the tests regress.
