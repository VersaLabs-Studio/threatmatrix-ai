## 🔴 Critical Finding: War Room Uses HTTP Polling, Not WebSocket

**Discovery:** War Room console shows regular updates every ~3-5 seconds via HTTP API calls (not WebSocket). The `totalAlerts` count stays at 344 (not incrementing), suggesting `setInterval`-based polling rather than WebSocket event-driven pushes.

**Impact:** Subtask 2.4 (Flow Push Test) cannot pass until WebSocket is properly implemented.

**Root Cause Analysis:**
- `useFlows` hook polls REST API every 3 seconds (`setInterval`)
- `useAlerts` hook polls REST API every 5 seconds (`setInterval`)
- `useWebSocket` hook EXISTS and properly subscribes to `flows:live`, `alerts:live`, `system:status`, `ml:live` channels
- But components never consume the WebSocket event state — they only use the REST polling data

**Solution:** Full refactor to WS-first architecture. Components will consume typed event state from `useWebSocket` hook, replacing `setInterval` polling as the primary data source.

### Step 1: Add FlowEvent Accumulation to useWebSocket Hook

**File:** `frontend/hooks/useWebSocket.ts`

The hook currently only stores the LAST flow event (`lastFlowEvent`). For the War Room metrics (Packets/s, Flows, Anomaly %), we need to accumulate flow events over time.

**Changes:**
1. Add `flowEvents` array to accumulate recent flows (max 1000)
2. Add `flowStats` derived state (total_packets, total_flows, anomaly_count)
3. Update `lastFlowEvent` listener to append to `flowEvents` and recalculate stats
4. Return `{ ..., flowStats, flowEvents }` alongside existing state

```typescript
const [flowEvents, setFlowEvents] = useState<FlowEvent[]>([]);
const [flowStats, setFlowStats] = useState<{
  total_packets: number;
  total_flows: number;
  anomaly_count: number;
}>({ total_packets: 0, total_flows: 0, anomaly_count: 0 });

// In subscribe callback for FLOWS channel:
const unsubFn = wsClient.subscribe(WS_CHANNELS.FLOWS, (d) => {
  const flow = d as FlowEvent;
  setLastFlowEvent(flow);
  setFlowEvents((prev) => {
    const updated = [flow, ...prev].slice(0, 1000);
    return updated;
  });
  setFlowStats((prev) => ({
    total_packets: prev.total_packets + 1,  // simplified, real impl needs bytes tracking
    total_flows: prev.total_flows + 1,
    anomaly_count: prev.anomaly_count + (flow.is_anomaly ? 1 : 0),
  }));
});
```

### Task 4: Remaining Phase 2 Tasks

#### 4.1: Detection Latency Logging

**File:** `backend/ml/inference/worker.py`

**Required Changes:**
1. Add timestamps at flow receipt, scoring completion, and alert creation
2. Calculate end-to-end latency: `t_alert_created - t_flow_received`
3. Log per-alert latency with breakdown (IF, RF, AE, publish times)
4. Publish latency metrics to Redis `ml:metrics` channel

**Implementation:**
```python
# In _process_message():
start = time.time()

# After scoring:
t_scored = time.time()

# In _create_alert():
t_alert_created = time.time()
latency_ms = (t_alert_created - start) * 1000

logger.info("[Worker] Alert created in %.1fms (IF:%.1f, RF:%.1f, AE:%.1f, publish:%.1f)",
    latency_ms, if_time_ms, rf_time_ms, ae_time_ms, publish_time_ms)

# Publish to Redis
await self._publisher.publish("ml:metrics", json.dumps({
    "event": "latency",
    "payload": {
        "flow_id": flow_id,
        "latency_ms": round(latency_ms, 2),
        "inference_ms": round((t_scored - start) * 1000, 2),
        "timestamp": datetime.now(timezone.utc).isoformat(),
    }
}))
```
