# Plan: Fix Detection Latency Display on Frontend

## Problem Analysis
The Detection Latency Display widget in the War Room shows no data (all zeros) despite the ML worker publishing latency metrics to the `ml:metrics` Redis channel.

## Root Cause
1. **WebSocket Data Parsing Bug**: In `useWebSocket.ts`, the metrics subscription extracts `d.data` as MLMetricEvent, but the published message structure is `{"event": "latency", "payload": {...}}`. The actual metrics are in `d.data.payload`, not `d.data` directly.

2. **ML Worker Publishing**: Confirmed that `worker.py` publishes to "ml:metrics" with the correct structure.

3. **WebSocket Backend Forwarding**: `websocket.py` subscribes to `CHANNEL_ML_METRICS` and broadcasts messages correctly.

4. **Widget Implementation**: `LatencyWidget.tsx` exists and is properly integrated in `war-room/page.tsx`.

## Solution Steps
1. **Fix WebSocket Hook Data Extraction**
   - Modify `useWebSocket.ts` to extract metrics from `d.data.payload` instead of `d.data`
   - Ensure the extracted data matches the `MLMetricEvent` interface

2. **Add Debug Logging**
   - Add console.log in `LatencyWidget.tsx` to verify if `lastMetricEvent` is received
   - Ensure WebSocket connection is established and metrics channel is subscribed

3. **Test Data Flow**
   - Verify ML worker is actively publishing metrics (check Docker logs)
   - Confirm Redis pub/sub is working (`redis-cli PUBSUB CHANNELS`)
   - Test with a simulated anomaly to trigger metric publishing

4. **Widget Rendering Verification**
   - Ensure Recharts is properly imported and rendering
   - Check for any CSS issues preventing display
   - Verify the widget is positioned correctly in the War Room grid
