# Plan: Refactor Notification Audio + Visual Alerts (Task 1)

## Problem Analysis
Current implementation shows toasts for all severities (including MEDIUM/LOW from anomaly events), displays up to 5 toasts simultaneously, and may not gracefully show alert details. The user wants strict filtering to CRITICAL/HIGH only, limited to 3 notifications max, with proper timeouts and detailed display.

## Current Issues
1. **Over-notification**: Anomaly events trigger toasts for MEDIUM+ severity (score >=0.50), leading to many MEDIUM alerts.
2. **Too many toasts**: Keeps up to 5 toasts, potentially overwhelming during attack simulations.
3. **Detail display**: While toasts include scores/IPs/ML breakdown, they may not feel "graceful" or focused on key alert details.
4. **No filtering**: No strict enforcement of CRITICAL/HIGH only for all notification types.

## Proposed Solution

### 1. Filter Notifications Strictly
- Modify `AppShell.tsx` to only create toasts for CRITICAL and HIGH severity events.
- For anomaly events: Remove the MEDIUM+ condition and only add if severity is 'critical' or 'high'.
- For alert events: Ensure only CRITICAL/HIGH are processed (assuming backend filters, but add guard).

### 2. Limit Concurrent Notifications
- Change toast array limit from 5 to 3: `setToasts((prev) => [toast, ...prev].slice(0, 3));`
- This reduces visual clutter during high-volume alerts.

### 3. Enhance Alert Detail Display
- Update toast message to be more informative: Include category, source IP, and key score.
- Ensure alertId is present for navigation to details.
- For anomalies without alertId, add fallback to show more context (e.g., flow details if available).
- Adjust layout in `NotificationToast.tsx` to prioritize key details: Category, IPs, Score, with optional ML breakdown.

### 4. Maintain Audio/Visual Features
- Keep Web Audio API alarms for CRITICAL (siren) and HIGH (beep).
- Retain CriticalOverlay for CRITICAL alerts with category and srcIp.
- Ensure timeouts: Critical 10s, High 8s.

### 5. Testing and Validation
- Test with attack simulations to verify only CRITICAL/HIGH toasts appear.
- Confirm max 3 toasts during flood scenarios.
- Verify alert details include category, IPs, and scores prominently.
- Ensure no MEDIUM/LOW notifications.

## Implementation Steps
1. **Modify AppShell.tsx**:
   - Update anomaly event handler: Remove `if (score < 0.50) return;` and add severity check.
   - Change toast limit to 3.
   - Enhance toast message for better details.

2. **Update NotificationToast.tsx**:
   - Adjust layout to emphasize alert details.
   - Ensure responsive design for details.

3. **Test on VPS**:
   - Deploy changes, run attack simulation, verify behavior.

4. **Debug and Iterate**:
   - If details still not graceful, add more structured display (e.g., bullet points for key info).

## Acceptance Criteria
- [ ] Only CRITICAL and HIGH severity alerts trigger notifications.
- [ ] Max 3 concurrent toasts displayed.
- [ ] Each notification shows key details: Category, Source IP, Score, with link to full details.
- [ ] Audio alarms work: Siren for CRITICAL, beep for HIGH.
- [ ] CriticalOverlay appears for CRITICAL with attack info.
- [ ] No distracting MEDIUM/LOW notifications during normal or attack traffic.
