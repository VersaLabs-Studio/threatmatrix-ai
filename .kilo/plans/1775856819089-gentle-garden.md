# Plan: Fix AI Analyst Missing System Context

## Problem
AI Analyst shows "Insufficient data to complete the analysis" when asked about recent anomalies, even though the system has alerts.

## Root Causes Identified
1. **Alert Engine** (`alert_engine.py:146`) hardcodes `ml_model = "ensemble"` and does NOT persist the actual `rf_label` from the worker to the database
2. **LLM API** (`llm.py:157-159`) has incorrect field mappings:
   - `rf_label` uses `alert.get("ml_model")` which is always "ensemble"
   - `rf_confidence` uses `alert.get("confidence")` which is actually the composite score
3. **Database schema** missing the `rf_label` column entirely
4. **Context injection** (`llm.py:105`) references non-existent field `a.get('ml_model')` instead of the actual RF data

## Implementation Steps

### 1. Update Database Schema
- Add `rf_label VARCHAR(50)` column to `alerts` table
- Optional: Add migration script for existing databases

### 2. Fix Alert Engine Field Mapping (`alert_engine.py`)
- Remove hardcoded `ml_model = "ensemble"`
- Map `rf_label` from payload to database column
- Map `rf_confidence` correctly from payload
- Keep `ml_model` as "ensemble" for backward compatibility

### 3. Fix LLM API Endpoint Field Mappings (`llm.py`)
- Update `get_alert_by_id` response mapping to use actual `rf_label` column
- Update context injection loop to pull correct RF fields
- Fix the composite score / confidence mapping

### 4. Verify Worker Data Flow
- Confirm worker publishes correct fields: `rf_label`, `rf_confidence`
- Confirm all downstream services receive correct data

### 5. Add Fallback Logic
- Add graceful fallback when old alerts don't have rf_label
- Maintain backward compatibility with existing alert data

## Files to Modify
1. `backend/app/services/alert_engine.py` - Fix field persistence
2. `backend/app/api/v1/llm.py` - Fix field mappings
3. Database migration (create new migration script)
4. Optional: Update alert model in `app/models/alert.py`
