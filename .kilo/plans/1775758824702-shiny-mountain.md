## Critical Issue: All Alerts Are MEDIUM Severity

### Symptom
```json
{
    "total": 1482,
    "by_severity": {
        "critical": 0,
        "high": 0,
        "medium": 1482,
        "low": 0,
        "info": 0
    }
}
```

### Root Cause

The ML ensemble scoring pipeline produces composite scores that cluster tightly around **0.50-0.52**:

```
composite = 0.30 × IF_score + 0.45 × RF_attack_conf + 0.25 × AE_score
```

**Why scores cluster at 0.50:**

1. **Isolation Forest** (contamination=0.10): Flags ~10% of live traffic as anomalous, but with moderate scores (0.5-0.7). The model was trained on NSL-KDD data which has very different feature distributions than real internet traffic.

2. **Random Forest**: Classifies most live traffic as "normal" with high confidence. The RF_attack_conf (1.0 - P(normal)) is very low (0.1-0.3) for live traffic because the model was trained on labeled attack data.

3. **Autoencoder**: Has moderate reconstruction error for live traffic (0.4-0.6) because live traffic is noisy but not necessarily anomalous.

**The weighted average consistently lands at 0.50-0.52**, which barely crosses the MEDIUM threshold (0.50). The thresholds were designed for NSL-KDD score distributions, not live traffic.

### What Was Attempted (Did NOT Work)

1. **Threshold lowering** (0.90/0.75/0.50/0.30 → 0.65/0.55/0.45/0.35) — This was committed but the worker was NOT rebuilt with the new thresholds. The worker still uses the old threshold values from the running container.

2. **Attack simulation scripts** — Fixed API key mismatch (`total_alerts` → `total`), added optional API_URL parameter, added local attack script. But attacks don't generate diverse severity because the ML model produces the same score regardless of attack type.

3. **Capture engine optimization** — Fixed O(n²) bytes concat, capped IAT list, added loopback capture, reduced timeouts (30s→10s active, 120s→15s idle). This improved capture rate from 11.3 pps to ~50 pps, but does NOT affect ML scoring.

### What Needs to Happen Next

**Option A: Retrain models on live traffic (best long-term fix)**
- Collect labeled flows from live traffic (mark known attacks as anomalous)
- Retrain IF, RF, AE models
- This will take significant time and labeled data

**Option B: Adjust thresholds aggressively + add attack-specific score boosting (quick fix)**
- Set thresholds much lower: critical=0.55, high=0.50, medium=0.45, low=0.40
- Add attack-type-specific score multipliers in the ensemble scorer:
  - If RF label is "dos" → multiply composite by 1.3 (DDoS should be critical)
  - If RF label is "probe" → multiply composite by 1.15 (port scan should be high)
  - If RF label is "r2l" → multiply composite by 1.2 (brute force should be high)
  - If RF label is "u2r" → multiply composite by 1.4 (privilege escalation should be critical)
- This is a pragmatic approach for demo purposes

**Option C: Hybrid — threshold calibration + attack simulation via PCAP injection**
- Use pre-recorded PCAP files of known attacks (NSL-KDD test set)
- Replay PCAPs through the capture engine
- Since the models were trained on NSL-KDD features, PCAP replay should produce higher scores
- This is the most reliable approach for demo

**Recommended: Option B for immediate fix, Option C for reliable demo**
